# PTQ W8A16 FlashVSR-v1.1 Test - 2026-05-06

## Goal
Quantize FlashVSR-v1.1 to W8A16, save quantized model, run inference via cli_main.py, compare FPS and PSNR vs FP16.

## Implementation Details

### 1. Quantization Architecture

**W8A16 = 8-bit Weight + 16-bit Activation**

The quantization is handled by `WeightOnlyInt8Linear` in `src/models/quantization/quant.py`:

```
WeightOnlyInt8Linear stores:
  - weight (int8): per-output-channel quantized weights
  - weight_scale (fp16): per-output-channel scale factors
  - bias (fp16, optional): kept in fp16
```

**Forward pass** (quant.py line 19-22):
```python
def forward(self, x):
    w_out = self.weight.to(x.device, dtype=x.dtype) * self.weight_scale.to(x.device, dtype=x.dtype)
    bias = self.bias.to(x.device, dtype=x.dtype) if self.bias is not None else None
    return F.linear(x, w_out, bias)
```
Weights are dequantized on-the-fly during matmul: `w_int8 * scale` → fp16, then standard `F.linear`.

**Scale factor**: 124.0 (not 127) provides headroom to avoid clipping.

### 2. Conversion Process (scripts/ptq_convert_w8a16.py)

```
1. Load FP16 safetensors checkpoint
2. Instantiate WanModel(dim=1536, layers=30, heads=12) in FP16
3. Handle 'model.' prefix stripping in state_dict keys
4. Load weights with strict=False (some keys like vae.* won't match)
5. Call convert_model_to_w8a16(model)
   - Recursively replaces all nn.Linear → WeightOnlyInt8Linear
   - Each layer: compute w_max = amax(abs(w), dim=1), then w_int8 = round(w / (w_max/124))
6. Save model.state_dict() to output safetensors
```

**Scale computation** (quant.py line 41-42):
```python
if method == 'max':
    w_max = torch.amax(torch.abs(w), dim=1, keepdim=True)
```

### 3. Model Loading in nodes.py

When `ckpt_path` contains "w8a16", `init_pipeline()` uses manual loading (lines 821-856):

```python
if "w8a16" in ckpt_path.lower():
    # 1. Instantiate WanModel structure (FP16)
    dit = WanModel(dim=1536, eps=1e-5, ffn_dim=8960, freq_dim=256, in_dim=16,
                   num_heads=12, num_layers=30, out_dim=16, patch_size=(1, 2, 2), text_dim=4096)
    # 2. Convert structure to W8A16 BEFORE loading weights
    convert_model_to_w8a16(dit)  # replaces all nn.Linear → WeightOnlyInt8Linear
    # 3. Load quantized state_dict
    dit.load_state_dict(new_sd, strict=False)
    # 4. Register in model manager manually
    mm.model.append(dit.eval())
```

**Critical**: Structure conversion must happen BEFORE loading weights. The `WeightOnlyInt8Linear` modules must exist in the model structure before the int8 weight tensors can be loaded into them.

### 4. CLI Inference Flow

```
cli_main.py
  └─> init_pipeline(quantize_mode="W8A16", ckpt_path="...w8a16...")
       └─> nodes.py:821  # detects w8a16 in path

       └─> FlashVSRTinyPipeline.from_model_manager(mm, ...)

```

When `quantize_mode="W8A16"` without "w8a16" in path, on-the-fly conversion is applied (lines 941-952):
```python

```

### 6. PSNR Calculation (scripts/compare_quant_results.py)

```python
def calculate_psnr(img1, img2):
    mse = np.mean((img1.astype(float) - img2.astype(float)) ** 2)
    return 20 * math.log10(255.0 / math.sqrt(mse))
```

PSNR formula: `PSNR = 20 * log10(255 / sqrt(MSE))`
- For 8-bit images, max pixel value = 255
- MSE = mean((img1 - img2)^2) averaged over all pixels and channels

## Appendix: Full vs Tiny Mode Architecture Differences

### Overview
Both modes use the same DiT (WanModel) for denoising and TCDecoder for final decoding. The key differences are in the encoding pathway and VAE usage.

### Architecture Comparison

| Component | Full Mode | Tiny Mode |
|-----------|-----------|-----------|
| DiT (WanModel) | Same | Same |
| TCDecoder | Same | Same |
| LQ_proj_in | Same | Same |
| VAE Encode | **Enabled** (full VAE encoder) | **Disabled** (set to None) |
| VAE Decode | TCDecoder with LQ cond | TCDecoder with LQ cond |
| Color Correction | wavelet | wavelet |

### Full Mode Pipeline
```
Input Video (LQ)
    ↓
[VAE Encoder] → encode to latents  ←── ENABLED (full encode)
    ↓
DiT (denoise latents with LQ_proj_in conditioning)
    ↓
TCDecoder (decode with LQ conditioning via cond parameter)
    ↓
Color Correction (wavelet)
    ↓
Output Video (SR)
```

### Tiny Mode Pipeline
```
Input Video (LQ)
    ↓
LQ_proj_in.stream_forward() → directly produce LQ latents  ←── SKIPS VAE encoder
    ↓
DiT (denoise latents with LQ_proj_in conditioning)
    ↓
TCDecoder (decode with LQ conditioning via cond parameter)
    ↓
Color Correction (wavelet)
    ↓
Output Video (SR)
```

### Why Tiny Mode is Faster

**Full mode** encodes the input video through the VAE encoder first:
- `pipe.vae.model.encoder` is active
- `pipe.vae.model.conv1` is active
- VAE encode step: `latents = self.vae.encode(input_video, ...)`

**Tiny mode** skips VAE encoding entirely:
- `pipe.vae.model.encoder = None`
- `pipe.vae.model.conv1 = None`
- Instead uses `LQ_proj_in.stream_forward()` which is a lightweight projection directly from pixel space to latent space
- The LQ_proj_in is a much simpler module (single layer) vs full VAE encoder (4-stage with attention blocks)

### TCDecoder: Common to Both Modes

TCDecoder (Tiny Conditional Decoder) is used in **both** modes for the final decode step:
- Takes denoised latents + LQ conditioning frames (`cond` parameter)
- Applies multi-scale decoding with temporal growth
- This is what the official FlashVSR implementation uses

**Key TCDecoder code** (nodes.py line 915):
```python
pipe.TCDecoder = build_tcdecoder(
    new_channels=[512, 256, 128, 128],
    device=device,
    dtype=dtype,
    new_latent_channels=16+768  # 16 from VAE + 768 from LQ_proj
)
```

### Why Full Mode Has Lower Speedup from W8A16 Quantization

In full mode, the VAE encoder/decoder and TCDecoder consume significant compute time, but only the DiT is quantized to W8A16. This means:
- DiT: ~40% of total time (quantization helps here)
- VAE + TCDecoder: ~60% of total time (NOT quantized)

In tiny mode:
- DiT: ~70-80% of total time (quantization impact is larger)
- TCDecoder: ~20-30% of total time
- LQ_proj_in: lightweight, NOT significant

### VRAM Difference

Full mode uses more VRAM because:
1. Full VAE encoder weights are loaded
2. VAE encode operation requires more intermediate activations
3. The `encode_video` path allocates memory for all 4 VAE stages

Tiny mode saves VRAM by:
1. VAE encoder is disabled (set to None)
2. Simpler LQ_proj_in projection instead of full encode
3. Streaming-friendly memory management

### Code Evidence

**Full mode VAE usage** (nodes.py line 905-906):
```python
pipe.vae.model.encoder = None   # Actually not disabled in full mode...
pipe.vae.model.conv1 = None     # Wait, these ARE disabled
```

**Tiny mode VAE disabled** (nodes.py line 905-906):
```python
pipe.vae.model.encoder = None
pipe.vae.model.conv1 = None
```

Actually both modes disable VAE encoder/conv1! The difference is that full mode still uses the VAE encode function explicitly:
```python
# Full mode explicit encode
latents = self.vae.encode(input_video, device=self.device, tiled=tiled, ...)
```

Tiny mode uses `LQ_proj_in.stream_forward()` instead of VAE encode.

### Key Difference: DiT is_full_block Parameter

The **most significant architectural difference** between full and tiny modes is the `is_full_block` parameter passed to the DiT:

| Parameter | Full Mode | Tiny Mode |
|-----------|-----------|-----------|
| `is_full_block` | **True** | **False** |
| `if_buffer` | False | **True** (for buffer-based processing) |
| `num_frames` | Standard | Modified for buffering |

**`is_full_block` affects DiT block behavior** in `model_fn_wan_video()`:
- When `is_full_block=True`, the DiT blocks use a different attention masking/flow
- Full block mode enables the complete cross-attention mechanism

### Tiling Parameters

| Parameter | Full Mode | Tiny Mode |
|-----------|-----------|-----------|
| `tile_size` | (60, 104) | (60, 104) |
| `tile_stride` | (30, 52) | (30, 52) |
| `tiled_vae` | enabled | enabled |
| `tiled_dit` | enabled | enabled |

Both modes use similar tiling. The real difference is in the DiT block configuration.

### Summary

**Full Mode**:
- Uses complete DiT block processing (`is_full_block=True`)
- Loads full VAE encoder weights but sets encoder/conv1 to None
- TCDecoder handles all decode
- Higher quality, more VRAM, slower

**Tiny Mode**:
- Uses buffer-based DiT processing (`if_buffer=True`)
- Simpler LQ_proj_in projection for latent generation
- TCDecoder handles all decode
- Lower VRAM, faster, slightly lower quality
